## 关键决策

**选择规则引擎而不是 LLM 做消息分类**

- 原因：规则引擎速度快、可调试、无 API 成本；群消息有很强的信号特征（文件扩展名、问号、@所有人）
- 分类优先级：submission > announcement > question > answer > resource > discussion > social
- 如果重来：对于实际部署可以加 LLM 精分类层（hybrid 模式），Bronze 级规则 + Silver 级 LLM

**Q&A 匹配策略：reply_to + 时间窗口双通道**

- reply_to 精确匹配回复链；时间窗口（10分钟内、不同发送者）捕获没有明确回复的对话
- 质量评分基于答案长度 + 关键词信号，过滤低质量回答

## 困难与突破

**问题：** 真实群只有 55 条消息（群昨晚建立），Q&A 极少  
**解决：** 创建 26 条测试数据（`tests/fixtures/sample_messages.jsonl`），涵盖真实场景（SSH key 配置、GitHub Pages 延迟、wechat-cli 报错等），达到 9 个高质量 Q&A 对

**问题：** 提交检测中 LEVEL_REF 正则有 DeprecationWarning  
**解决：** 标记为已知问题，不影响功能，待后续重构时用 `(?:level\s*|L([1-4]))` 修复
